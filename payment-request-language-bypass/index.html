Payment Request API Language Fingerprint Bypass

Setup (to test the protection):
  1. Open brave://settings/shields (or privacy) and enable "Prevent sites from fingerprinting me based on my language preferences" (this is on by default with Shields fingerprinting protection).
  2. Open brave://settings/languages and add several non-English preferred languages (e.g. French, German, Japanese), so your full list is distinctive.
  3. Reload this page and click Run Test.

This page is on test-website-a.pages.dev. The payment-method endpoints live on the unrelated third-party origin test-website-b.pages.dev, which is where your full language list leaks to.

How it works / why this is a bypass 
const request = new PaymentRequest(
  [{ supportedMethods: 'https://this-site/.../pmh/<token>' }],
  { total: { label: 'total', amount: { value: '0', currency: 'USD' } } }
);
request.show();   // browser HEAD-fetches the manifest URL

Brave reduces Accept-Language to a single language inside BraveProxyingURLLoaderFactory (the proxy installed for renderer/frame network factories via WillCreateURLLoaderFactory). A normal fetch() from this page goes through that proxy and is reduced.

But PaymentManifestDownloader fetches the manifest using the browser-process factory (GetURLLoaderFactoryForBrowserProcess()). The renderer factory is only used when the feature kPaymentRequestUseRendererUrlLoader is enabled, and it is disabled by default. The browser-process factory is not wrapped by Brave's proxy, so the reduction never runs and the full Accept-Language list leaks — here, to an unrelated third-party origin (test-website-b.pages.dev) that the merchant page simply names as its payment method.

Verdict: if the Payment-manifest row below shows more languages than the normal fetch() row, the protection was bypassed.
